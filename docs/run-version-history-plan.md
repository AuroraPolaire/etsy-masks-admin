# Internal Run Version History Plan

## Goal

Add a per-run version control system so every important stage can be restored without losing
generated files. A user should be able to move back to a known-good point after a failed generation,
bad prompt change, accidental deletion, or stale marketing pass.

This is separate from the existing saved-runs list:

- A saved run is the project container.
- A history revision is an immutable checkpoint inside one saved run.
- Restoring a revision updates the current run state, but never deletes the old state.

## Product Principles

- Generated bytes must be recoverable once the app says they are saved.
- Restore must be non-destructive. The app creates a safety checkpoint before any rollback.
- History is scoped to the active run only. Assets from another run must never appear in the current
  run history.
- The feature should be visible on the main workflow, not buried in Cloud settings.
- The user should not need to understand D1, R2, or file manifests. The UI should speak in workflow
  stages, file counts, thumbnails, and plain restore choices.

## UX Design

### Main Workflow Entry

Add a compact History control near the workflow status area:

- Current save state: `Saved`, `Saving...`, `3 files not saved yet`, or `Cloud unavailable`.
- Last checkpoint label and time.
- A `History` button that opens a right-side drawer.
- A `Save checkpoint` secondary action for manual labels such as "Before redoing slogans".

Keep the Cloud page focused on saved runs. A restored run can still show checkpoint count in the
saved-runs table, but the history workflow belongs on the main page where the user is generating
and approving work.

### History Drawer

Use a two-column drawer on desktop and a single-column stacked drawer on mobile.

Timeline column:

- Group revisions by workflow stage: Brief, Masks, Approval, Coloring pages, Marketing, Export.
- Show an icon per revision type: auto checkpoint, manual checkpoint, restore point, generation
  checkpoint, final export.
- Show label, timestamp, file count, and a small thumbnail strip.
- Add filters: `All`, `Manual`, `Generated files`, `Restores`, `Marketing`.
- Add a search input for labels and stage names.

Details column:

- Revision summary: title, theme, subjects, settings, and file counts by variant.
- Thumbnail strip for changed files and final assets.
- Difference chips such as `+3 masks`, `2 approvals changed`, `marketing settings changed`.
- Stale indicators when final marketing assets were generated from older masks.
- Actions:
  - `Preview this checkpoint`
  - `Restore whole run`
  - `Restore project text/settings only`
  - `Restore files only` in a later phase if the first restore UX proves stable.

### Restore Flow

The restore dialog should make the safety model obvious:

1. Show what will change: project fields, subjects, approvals, files, and marketing assets.
2. Explain that the current state will be saved as a checkpoint first.
3. Require a clear `Restore checkpoint` confirmation button.
4. After restore, show a toast with `Restored "Approved masks before marketing"` and an `Undo`
   shortcut that restores the safety checkpoint.

### Stage Jump Shortcuts

Add small contextual shortcuts once history exists:

- In the Masks step: `Jump to last approved masks`.
- In Marketing: `Back to before marketing generation`.
- In Export: `Back to last export-ready checkpoint`.

These should open the same restore dialog rather than performing immediate changes.

## Checkpoint Behavior

### Automatic Checkpoints

Create an auto checkpoint after these milestones:

- Product brief generated or significantly edited.
- Subject list changes.
- Each generated mask file is uploaded.
- Bulk mask generation completes.
- One or more masks are approved or approval state changes.
- Coloring pages are generated.
- Marketing preview options are generated.
- A marketing option is finalized.
- Archive export succeeds.
- Before any restore.

For long generation batches, persist incrementally:

1. Ensure the run exists before the generation starts.
2. After each generated file is appended locally, upload that file immediately.
3. Write a small generation checkpoint after each successfully uploaded file or every few files.
4. Write a named milestone checkpoint when the batch completes.

This prevents losing the first generated assets if a later image call, browser tab, or network
request fails.

### Manual Checkpoints

Add `Save checkpoint` in the History control and drawer. Manual checkpoints should be pinned by
default and excluded from automatic retention cleanup.

### Retention

Keep all manual checkpoints, all restore safety checkpoints, and the newest export checkpoint.
For automatic checkpoints, keep at least the newest 100 per run, then collapse older high-frequency
generation checkpoints into milestone checkpoints. Do not delete any binary object that is still
referenced by the current run or any retained revision.

## Backend Design

### Current Constraint

The current `file_backups` row is the mutable current file state, and the R2 key can be overwritten
or deleted by normal file sync. A revision cannot safely point only at the current `file_backups`
row or R2 key because a later edit could remove the bytes.

The version system needs immutable file objects.

### D1 Schema

Add immutable file objects and run revisions:

```sql
CREATE TABLE IF NOT EXISTS file_objects (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  project_id TEXT NOT NULL,
  file_id TEXT NOT NULL,
  r2_key TEXT NOT NULL,
  thumbnail_r2_key TEXT,
  name TEXT NOT NULL,
  size INTEGER NOT NULL,
  type TEXT NOT NULL,
  sha256 TEXT,
  created_at TEXT NOT NULL,
  FOREIGN KEY (run_id) REFERENCES project_runs(id) ON DELETE CASCADE
);

ALTER TABLE file_backups ADD COLUMN object_id TEXT;

CREATE TABLE IF NOT EXISTS run_revisions (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  project_id TEXT NOT NULL,
  parent_revision_id TEXT,
  sequence_number INTEGER NOT NULL,
  stage TEXT NOT NULL,
  kind TEXT NOT NULL,
  label TEXT NOT NULL,
  description TEXT,
  project_json TEXT NOT NULL,
  file_manifest_json TEXT NOT NULL,
  change_summary_json TEXT,
  thumbnail_file_id TEXT,
  is_manual INTEGER NOT NULL DEFAULT 0,
  is_pinned INTEGER NOT NULL DEFAULT 0,
  restored_from_revision_id TEXT,
  created_at TEXT NOT NULL,
  FOREIGN KEY (run_id) REFERENCES project_runs(id) ON DELETE CASCADE
);

CREATE UNIQUE INDEX IF NOT EXISTS run_revisions_run_sequence_idx
  ON run_revisions(run_id, sequence_number);

CREATE INDEX IF NOT EXISTS run_revisions_run_created_at_idx
  ON run_revisions(run_id, created_at DESC);

CREATE INDEX IF NOT EXISTS file_objects_run_file_idx
  ON file_objects(run_id, file_id);
```

`file_manifest_json` stores the exact file rows for that revision, including metadata, review
state, marketing metadata, image dimensions, and `object_id`. It should be enough to reconstruct
`file_backups` without reading current mutable rows.

### File Storage Invariant

File uploads should create a new immutable object:

- R2 key shape: `runs/{runId}/objects/{objectId}/{safeName}`.
- `file_backups.object_id` points to the current object for the logical file id.
- Revision manifests point to `object_id`, not only `file_id`.
- Updating a file creates a new object and moves the current pointer.
- Deleting a current file removes the `file_backups` row but does not delete the object if any
  revision references it.
- Garbage collection deletes only objects that are not referenced by `file_backups` or retained
  `run_revisions`.

This is the core guarantee that generated data is not lost.

### API Endpoints

Add endpoints under the existing run API:

- `GET /api/runs/:runId/revisions`
  - Returns revision summaries for the timeline.
- `POST /api/runs/:runId/revisions`
  - Creates a checkpoint from the current `project_runs` row and current `file_backups` rows.
  - Body: `stage`, `kind`, `label`, `description`, `isManual`, optional `changeSummary`.
- `GET /api/runs/:runId/revisions/:revisionId`
  - Returns full checkpoint details and manifest.
- `PATCH /api/runs/:runId/revisions/:revisionId`
  - Updates label, description, pinned state.
- `POST /api/runs/:runId/revisions/:revisionId/restore`
  - Creates a safety checkpoint for current state.
  - Replaces `project_runs.project_json`.
  - Rebuilds current `file_backups` rows from the selected manifest.
  - Does not delete referenced R2 objects.
  - Returns the restored run snapshot plus safety revision id.

All endpoints must verify that the revision belongs to the requested `runId` and `project_id`.

### Sync Integration

Extend the current backend sync flow:

- After successful `syncRunToBackend`, create a checkpoint when the action includes a checkpoint
  reason.
- Generation hooks should pass explicit checkpoint reasons rather than relying only on generic
  autosave.
- A failed checkpoint should not block the UI forever, but it should show a clear warning that the
  latest generated files are not yet in history.
- Retry checkpoint creation in the same queue used for draft autosave.

## Data Loss Prevention

The app should make these guarantees:

- A generated file is uploaded as soon as it is created, not only at the end of a workflow step.
- Each upload creates an immutable object before it updates the current file pointer.
- A checkpoint references immutable object ids.
- Restore creates a safety checkpoint first.
- Current file deletion never deletes bytes referenced by history.
- Clearing local browser state cannot delete cloud revisions.
- A run restore only loads data from the selected run id.
- Cloud save indicators distinguish `local only`, `uploaded`, and `checkpointed`.

## Test Plan

### Worker Tests

- Creating a revision stores project JSON and file manifest for only that run.
- Revision manifests include `metadata_json`, thumbnails, review state, asset variant, source file
  id, and image dimensions.
- Uploading a changed file creates a new `file_objects` row instead of overwriting the old object.
- Deleting a current file does not delete an object referenced by any revision.
- Restoring a revision creates a safety checkpoint before changing the run.
- Restoring a revision rebuilds current file rows with the original object ids and metadata.
- A revision from run A cannot be read or restored through run B.
- `DELETE /api/runs/:runId` removes that run's revisions and objects without affecting another
  run.

### Frontend Unit Tests

- History summaries group revisions by stage and sort newest within stage.
- Checkpoint labels are generated from workflow action context.
- Generation actions request incremental checkpoints after uploaded files.
- The restore dialog shows changed project fields and file counts.
- Undo after restore targets the safety checkpoint.
- Cloud status distinguishes uploaded files from checkpointed files.

### Component And E2E Tests

- History control appears on the main workflow after a run exists.
- Manual checkpoint creation adds a timeline item without leaving the workflow.
- A generated mask is still recoverable after deletion and restore.
- A failed marketing generation can be rolled back to "before marketing".
- Restoring one run does not show marketing assets from another run.
- The app works when history is unavailable: generation remains usable, but the warning is visible.

## Open Questions

- Should automatic checkpoint retention be configurable, or is a fixed per-run limit enough?
- Should "files only" restore ship in the first restore PR, or wait until full-run restore is proven?
- Should export checkpoints include the generated ZIP metadata, or only the source files used to
  build the ZIP?
- Should history be available for purely local browser-only sessions, or only after the cloud run is
  reachable?
